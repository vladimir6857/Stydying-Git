Я учусь работать с Git Этот workflow обычно используется при совместной разработке в команде, где изменения вносятся в отдельные ветки, а затем предлагаются для слияния в основную ветку через механизм Pull Requests. 1. git clone : * Что это: Команда git clone используется для создания локальной копии удаленного репозитория. * Когда это нужно: Этот шаг выполняется только один раз, когда вы хотите получить копию существующего репозитория на свой компьютер. * Что происходит: Git скачивает все файлы, историю изменений (коммиты) и ветки из удаленного репозитория и сохраняет их в новую папку на вашем компьютере. - это адрес репозитория (например, на GitHub, GitLab или Bitbucket). 2. git checkout -b feature/new-feature: * Что это: Эта команда создает новую ветку с именем feature/new-feature и сразу же переключается на нее. * Когда это нужно: Когда вы начинаете работу над новой функцией, исправлением ошибки или любыми другими изменениями, которые не должны сразу же вноситься в основную ветку (main или master). * Что происходит: Git создает новую ветку, которая является копией текущей ветки (обычно main). Переключение на эту ветку позволяет вам вносить изменения, не затрагивая основную линию разработки. feature/new-feature - это просто пример названия ветки, вы можете выбрать любое подходящее имя. 3. (Редактировать файлы): * Что это: На этом этапе вы вносите изменения в файлы в своей локальной копии репозитория. * Когда это нужно: Когда вы разрабатываете новую функцию, исправляете ошибку, обновляете документацию или выполняете любую другую работу, связанную с изменением содержимого файлов. * Что происходит: Вы используете свой любимый текстовый редактор, IDE или другие инструменты для редактирования файлов. 4. git add .: * Что это: Эта команда добавляет все измененные и новые файлы в "staging area" (область подготовленных изменений). * Когда это нужно: После того, как вы внесли изменения в файлы, перед созданием коммита, вы должны указать Git, какие изменения вы хотите включить в следующий коммит. * Что происходит: Git запоминает, какие изменения вы добавили в staging area. Команда git add . добавляет все измененные и новые файлы в текущей директории и ее поддиректориях. Вы также можете добавлять отдельные файлы, используя git add . 5. git commit -m "Implement new feature": * Что это: Эта команда создает коммит, который является снимком текущего состояния staging area. * Когда это нужно: Когда вы закончили вносить логическую часть изменений и хотите сохранить их в истории репозитория. * Что происходит: Git берет все изменения, которые вы добавили в staging area, и создает новый коммит. -m "Implement new feature" добавляет сообщение к коммиту, описывающее внесенные изменения. Сообщение коммита должно быть информативным и описывать, что было сделано. 6. git push origin feature/new-feature: * Что это: Эта команда отправляет локальную ветку (feature/new-feature) на удаленный репозиторий (origin). * Когда это нужно: Когда вы хотите поделиться своими изменениями с другими членами команды или сохранить их на удаленном сервере. * Что происходит: Git загружает все коммиты из вашей локальной ветки на удаленный репозиторий. origin - это имя удаленного репозитория (обычно это репозиторий, с которого вы клонировали свой локальный репозиторий). 7. (Создать Pull Request): * Что это: Pull Request (PR) - это запрос на слияние вашей ветки (feature/new-feature) в основную ветку (main или master) на удаленном репозитории (например, GitHub, GitLab, Bitbucket). * Когда это нужно: После того, как вы отправили свою ветку на удаленный репозиторий, вы создаете Pull Request, чтобы сообщить другим членам команды, что вы закончили работу над функцией и готовы к слиянию. * Что происходит: Вы переходите на веб-интерфейс платформы (например, GitHub) и создаете Pull Request, выбирая свою ветку (feature/new-feature) в качестве исходной и основную ветку (main) в качестве целевой. Вы добавляете описание Pull Request, объясняя, что было сделано и почему. 8. (После одобрения, слияние): * Что это: Другие члены команды просматривают ваш Pull Request, проверяют ваш код, комментируют и предлагают изменения. После того, как все согласны с изменениями, Pull Request одобряется и сливается в основную ветку. * Когда это нужно: После того, как вы создали Pull Request. * Что происходит: Разработчики проверяют код, оставляют комментарии и предложения. После получения одобрения, один из разработчиков или владелец репозитория нажимает кнопку "Merge" (или аналог) на платформе (например, GitHub). Это действие объединяет изменения из вашей ветки feature/new-feature в целевую ветку (main). 9. git checkout main: * Что это: Эта команда переключает вашу локальную копию репозитория на основную ветку (main). * Когда это нужно: После того, как ваш Pull Request был слит в основную ветку, вы должны обновить свою локальную копию основной ветки. * Что происходит: Git переключает вашу рабочую директорию на состояние, соответствующее последнему коммиту в ветке main. 10. git pull origin main: * Что это: Эта команда скачивает последние изменения из удаленной ветки main и объединяет их с вашей локальной веткой main. * Когда это нужно: Чтобы получить последние изменения, внесенные другими членами команды, в вашу локальную копию основной ветки. * Что происходит: Git скачивает все новые коммиты и изменения из удаленной ветки main и применяет их к вашей локальной ветке main. 11. git branch -d feature/new-feature: * Что это: Эта команда удаляет локальную ветку feature/new-feature. * Когда это нужно: После того, как ваша ветка была успешно слита в основную ветку, и вы больше не нуждаетесь в ней локально. * Что происходит: Git удаляет ветку feature/new-feature из вашей локальной копии репозитория. Флаг -d означает "delete", и ветка будет удалена только если она была слита в другую ветку. Если вы не уверены, что ветка была слита, используйте git branch -D feature/new-feature (с большой буквой D), но будьте осторожны, так как это может привести к потере изменений. Этот workflow обеспечивает согласованность и контроль над изменениями в коде, особенно в командной разработке. Pull Requests позволяют проводить code review, обсуждать изменения и избегать конфликтов, что повышает качество кода и снижает риск ошибок.
